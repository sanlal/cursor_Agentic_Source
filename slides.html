The 4 ways to talk to Cursor

Rule of thumb: start at the smallest level that gets the job done. Tab finishes your line. Ctrl+K rewrites a block. Ctrl+L asks a question. Ctrl+I builds the feature.
This is the single most important slide for new users. Cursor has four interaction modes, in increasing order of how much work you delegate. Tab autocomplete - the killer feature. It predicts your next edit, even across files. Cmd-K - select code, describe a change, get a diff in place. Chat with Ctrl-L - sidebar Q&A; can apply edits if you ask. And Agent with Ctrl-I - autonomous, multi-file, uses tools. Beginners often jump straight to Agent for everything. Don't. Use the smallest tool that works: tab for a line, Cmd-K for a block, chat for a question, agent for a feature. You'll be 3x faster.

The @ symbol — feed Cursor context

Try this in Chat: "Compare @auth/login.ts with @auth/oauth.ts and pull shared logic into a util. Use @docs for NextAuth idioms."
No @codebase = AI guesses from open files only. Always reference what matters - quality jumps immediately.
The @ symbol is the second killer feature most beginners ignore. Cursor's AI is only as smart as the context you give it. @file pulls a single file. @folder pulls a directory. @code pulls a specific function or class. @docs lets you reference indexed library docs - React, Next.js, Stripe, hundreds of libraries. @web does a live web search. @git pulls a diff or commit history. @codebase does retrieval across your whole repo using embeddings. And @past-chat references an earlier session. Every prompt should start with one or two @-references. It is the difference between Cursor guessing and Cursor knowing.

Agent mode in depth

  • Reads any file, quotes back exact lines
  • Edits as diffs — accept / reject per chunk
  • Searches via grep/glob across the repo
  • Runs shell: npm test, git, installs
  • Checkpoints: rollback to any earlier state
  • Auto-run toggle: accept safe commands automatically
Try this: "Add a dark-mode toggle in @components/Header.tsx. Persist in localStorage. Respect prefers-color-scheme. Add a test."
Agent mode is the autonomous one. Open it with Ctrl-I. You give it a goal; it picks a tool, runs it, reads the output, picks the next tool. The four tools are read, edit, search, shell. Two safety nets you should know about: every edit is shown as a diff before you accept - per chunk, not per file. And every agent run creates a checkpoint, so you can rewind to any earlier state if it went sideways. There's an auto-run toggle that whitelists "safe" shell commands so the loop doesn't pause on every npm install - turn it on once you trust the model. The example prompt at the bottom is a real one I'd give the agent: notice the @-reference, the specific behaviour spec, and the "add a test" at the end - that's how to get production-quality output.

Plan Mode — design before code

Use Plan when: change touches >2 files · involves auth, payments, migrations · requirements are fuzzy · you'd otherwise ask 3+ clarifying questions.
Toggle the mode dropdown (Plan / Agent / Ask) at the top of the chat pane. Plan produces a written checklist you confirm before any edit happens.
Plan mode is Agent mode with one hand tied behind its back - and that's exactly the point. In Plan mode, the agent can read your code but can't edit anything. It does the research, asks you clarifying questions, then produces a written plan: which files it'll touch, what tests it'll add, what trade-offs it sees. You review the plan. If you like it, you confirm and it switches itself to Agent mode and executes. If you don't, you tweak the plan and re-run. The rule I follow: any change that touches more than two files, or anything in auth, payments, billing, or migrations - I always start in Plan mode. Ten minutes of planning saves an hour of undoing.

Rules & AGENTS.md

  • .cursor/rules/*.mdc — project-scoped instructions
  • AGENTS.md — portable, repo-root manifest
  • Read on every agent invocation — no per-prompt repeat
  • Encode standards: package mgr, test layout, naming, security
  • Use globs: frontmatter to scope rules per file type
  • Bonus: SKILL.md for reusable multi-step playbooks
Example rule: "Always use pnpm, never npm. Co-locate tests as *.test.ts. Public APIs need JSDoc. Never commit .env."
Once you start using Cursor at work, you'll get tired of writing the same instructions every prompt. "Use pnpm, not npm." "Tests live next to source." "We use Vitest, not Jest." Rules fix this. Drop a markdown file inside dot-cursor-slash-rules and Cursor reads it on every agent run. The .mdc extension lets you add frontmatter - globs that scope a rule to certain file types, or a description that lets the agent decide if the rule applies. AGENTS.md is a newer, cross-tool standard at the repo root that works in Cursor, Claude Code, and others. And SKILL.md files describe reusable workflows the agent picks up automatically. These three together turn a fresh repo into an agent-ready repo.

MCP, Hooks, Background

  • MCP servers: give the agent new tools
  • — Jira, Slack, Postgres, GitHub, Confluence...
  • Hooks (hooks.json): run scripts on agent events
  • — auto-format, auto-test, audit-log every edit
  • Background Agents: run in the cloud, on a branch
  • — perfect for slow refactors & multi-repo work
Power combo: MCP-Jira lets the agent pick up tickets; Background Agent implements; Hook posts the PR link back to the Jira ticket. Zero context-switching.
These three features turn Cursor from a code editor into an actual agent platform. MCP - Model Context Protocol - is an open spec from Anthropic. Any MCP server exposes tools the agent can call: Jira to read tickets, Postgres to query a database, GitHub to manage PRs. Hooks - hooks.json in dot-cursor - let you run shell scripts when certain agent events fire: auto-format on edit, auto-test on save, audit-log every accepted diff. Background Agents are the showstopper: kick off a long task, it runs in the cloud on its own branch in a clean sandbox, and you get a PR when it's done. You can fire off a refactor before lunch and review it after. Combined, these are how senior engineers ship 3x faster with Cursor.
The 3-step prompt: 1) "Write failing tests for slugify(str): spaces, accents, empties, max length." 2) "Now make the tests pass — minimum code only." 3) "Refactor for readability. Tests must stay green."
The single biggest quality multiplier when working with agents is test-driven development. The cycle is Red, Green, Refactor. Red - you ask the agent to write failing tests first. This is the magic step: those tests become a precise specification the AI has to satisfy. No more "looks right but doesn't work." Green - the agent writes the minimum code to pass. Because the tests exist, you know exactly when it's done. Refactor - the agent cleans up, with the tests as a safety net. The three example prompts at the bottom are the actual prompts I use - notice they're written one at a time, not as a single big request. Iterative prompts give the agent shorter feedback loops and dramatically better output.
